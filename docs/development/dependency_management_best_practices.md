# 依赖管理与环境稳定性最佳实践 (TiebaMecha)

本指南旨在总结如何避免因第三方库（如 Flet）版本更新导致的 API 不兼容问题，并确保开发环境与生产环境的一致性。

## 1. 核心问题回顾：语义化版本陷阱

在 `pyproject.toml` 中使用 `flet>=0.21.0` 时，系统会自动拉取满足条件的最高版本（如 `0.83.1`）。对于尚处于快速迭代期（0.x 版本）的框架，大版本跳跃通常伴随破坏性变更（Breaking Changes）：
- **API 结构调整：** 例如 `ft.icons` 从变量代理变为模块。
- **异步模型变更：** 例如 `FilePicker` 从事件驱动改为直接返回。
- **属性重命名：** 例如 `FilePickerResultEvent` 被移除或重命名。

## 2. 防御性方案建议

### 2.1 严格限制版本区间 (Version Constraints)
在 `pyproject.toml` 中，应为不稳定的依赖库设置明确的上限：
```toml
dependencies = [
    "flet>=0.21.0,<0.30.0",  # 仅允许 0.2x 系列，避免 0.8x 系列的 Breaking Changes
]
```

### 2.2 使用依赖锁定文件 (Lockfiles)
建议引入 **Poetry** 或 **Pipenv** 代替传统的 `pip`：
- **优点：** 自动生成 `poetry.lock`，记录所有依赖及其子依赖的精确版本和哈希值。
- **效果：** 确保每位开发者（或部署环境）拉取的代码环境 **100% 字节级一致**。

### 2.3 持续集成验证 (CI Monitoring)
利用 GitHub Actions 建立自动化流水线：
- 每次 Push 时在干净的 Docker 环境中全量安装依赖。
- 如果上游库更新导致项目挂掉，CI 会在合并代码前报警，而不是在本地开发时才发现。

### 2.4 版本快照习惯
在每次项目里程碑（如 v1.0, v2.0）发布前，执行环境快照：
```bash
pip freeze > requirements.lock
```
这能作为“救命稻草”，在极端情况下手动还原出当时测试通过的环境。
